## 2. Authentication & Session

### Session Cookie

- **Name**: `smartup_session` (httpOnly, secure, sameSite=lax, 7-day expiry)
- **Encoding**: base64-encoded JSON
- **Contents**:
  ```json
  {
    "email": "user@example.com",
    "api_key": "...",
    "api_secret": "...",
    "roles": ["Branch Manager", "Academics User"],
    "full_name": "John Doe",
    "allowed_companies": ["Smart Up Chullickal", "Smart Up Ernakulam"],
    "default_company": "Smart Up Chullickal",
    "instructor_name": null,
    "instructor_display_name": null,
    "allowed_batches": [],
    "default_batch": null
  }
  ```

### Login Flow

```
POST /api/auth/login { email, password }
  │
  ├─ 1. Verify via Frappe /api/method/login
  ├─ 2. Fetch User doc (admin auth) → extract roles from child table
  ├─ 3. Generate API key/secret via generate_keys method
  ├─ 4. Fetch User Permissions → build allowed_companies[], allowed_batches[]
  ├─ 5. Detect Instructor: Employee(user_id) → Instructor(employee) chain
  │     If instructor: prepend "Instructor" role + ensure "Academics User"
  ├─ 6. Create session cookie (base64 JSON)
  └─ 7. Return { email, full_name, roles, default_company }
```

### Role Priority (first match wins)

Director > Management > Branch Manager > HR Manager > Administrator > Instructor > Batch Coordinator > Teacher > Accounts User > Parent

### Auth Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/auth/login` | POST | Credentials → session + User |
| `/api/auth/me` | GET | Session → User (from cookie, no re-fetch) |
| `/api/auth/logout` | POST | Clear cookie + notify Frappe |
| `/api/auth/create-parent-user` | POST | Create Frappe User with "Parent" role + welcome email |
| `/api/auth/forgot-password` | POST | Generic message (no email leak) |

## 3. Fee Structure & Categories

### Hierarchy

```
Fee Category (template)
  └─ Fee Component (child table, reusable)
       └─ Fee Structure (per Program + Academic Year)
            ├─ custom_plan: "Basic" | "Intermediate" | "Advanced"
            ├─ custom_no_of_instalments: "1" | "4" | "6" | "8"
            └─ components[] (Fee Component rows)
                 └─ Fee Schedule (batch-level creation)
                      └─ Fees Record (per-student invoice)
```

### Fee Category DocType
- **Fields**: `category_name` (unique), `description`, `item` (→ Item)
- **Naming**: by `category_name`

### Fee Component DocType (Child Table)
- **Fields**: `fees_category` (→ Fee Category), `description`, `amount` (Currency), `item` (read-only, fetched from category), `discount` (%), `total` (computed)
- **Formula**: `total = amount × (1 - discount/100)`

### Fee Structure DocType
- **Naming**: `EDU-FST-.YYYY.-`
- **Key Fields**:
  - `program` (→ Program) — required
  - `academic_year` (→ Academic Year) — required
  - `components[]` (→ Fee Component) — required, min 1
  - `total_amount` (calculated)
  - `receivable_account` (→ Account) — required
  - `company` (→ Company/Branch)
- **Custom Fields**:
  - `custom_plan` — "Basic" | "Intermediate" | "Advanced"
  - `custom_no_of_instalments` — "1" | "4" | "6" | "8"
  - `custom_branch_abbr` — e.g., "SU ERV"
- **Submittable**: docstatus 0→1→2

### Fees DocType (Per-Student Invoice)
- **Naming**: `EDU-FEE-.YYYY.-`
- **Key Fields**:
  - `student` (→ Student) — required
  - `program_enrollment` (→ Program Enrollment) — required
  - `fee_structure` (→ Fee Structure) — required
  - `company` (→ Company/Branch) — required
  - `posting_date`, `due_date` — required
  - `components[]` (→ Fee Component)
  - `grand_total`, `outstanding_amount` (calculated)
  - `send_payment_request` (Check) — triggers email
  - `include_payment` (Check) — hidden, auto-submit flag
- **Permissions**: Academics User (CRUD), Accounts User (+submit), Accounts Manager (+cancel)

### Fee Schedule DocType (Batch-Level Generation)
- **Naming**: `EDU-FSH-.YYYY.-`
- **Purpose**: Generate Fees/Sales Orders for an entire Student Group at once
- **Key Fields**:
  - `fee_structure` (→ Fee Structure) — required
  - `student_groups[]` (→ Fee Schedule Student Group) — required
  - `send_email` (Check) — email payment requests to students
  - `status`: Draft → Order Pending → In Process → Order Created / Invoice Created / Failed
- **Links to**: Sales Invoice (fee_schedule), Sales Order (fee_schedule)

## 4. Sales Order Workflow

### Document Lifecycle

```
Draft (docstatus=0) ─── Submit ───▶ Submitted (docstatus=1)
        │                                    │
        │ edit/delete                         ├── Track per_billed %
        │                                    ├── Track per_delivered %
        ▼                                    │
    (discarded)                              ├── Create Sales Invoice
                                             │
                                             └── Cancel (docstatus=2)
```

### Sales Order Structure

| Field | Type | Notes |
|-------|------|-------|
| `customer` | Link → Customer | **Required**. Student's customer record |
| `transaction_date` | Date | **Required**. Order creation date |
| `delivery_date` | Date | Payment due/expected date |
| `company` | Link → Company | **Required**. Branch company |
| `items[]` | Child Table | Line items (tuition fees) |

## 5. Sales Invoice & Billing

### Creating Invoices from Sales Orders

Two methods available:

1. **`makeSalesInvoiceFromSO(soName)`** — Uses Frappe's native `make_sales_invoice` method
   - Returns mapped doc (NOT saved) with proper `so_detail` linkage per item
   - Only includes unbilled qty
   - Frontend can modify before saving

2. **`createInvoiceFromOrder(soName)`** — Full automation
   - Calls `makeSalesInvoiceFromSO` → inserts → submits
   - Returns completed Sales Invoice

### Sales Invoice Structure

| Field | Type | Notes |
|-------|------|-------|
| `customer` | Link → Customer | **Required** |
| `posting_date` | Date | **Required**. Invoice date |
| `due_date` | Date | Payment due date |
| `company` | Link → Company | **Required**. Branch |
| `items[]` | Child Table | With `sales_order` + `so_detail` back-refs |
| `grand_total` | Currency | Total billed |
| `outstanding_amount` | Currency | Unpaid balance |
| `status` | Select | Draft/Submitted/Paid/Unpaid/Overdue/Cancelled |

### Invoice Status Flow

```
Draft → Submitted → Unpaid ─── Payment ──▶ Paid
                       │                    
                       └── Past due_date ──▶ Overdue
```

### Frontend API Functions

| Function | Purpose |
|----------|---------|
| `getSalesInvoices(params?)` | List with filters (outstanding_only, company, customer, etc.) |
| `getSalesInvoice(name)` | Full detail with items + payments |
| `createSalesInvoice(payload)` | Create draft |
| `submitSalesInvoice(name)` | Submit |
| `cancelSalesInvoice(name)` | Cancel |

### Server-Side Fee Aggregation APIs

**`GET /api/fees/class-summary?company=...`**
- Admin-authenticated (server-side token)
- Groups Sales Invoices by `item_code` (class/program)
- Returns: `{ item_code, student_count, total_outstanding }` per class

**`GET /api/fees/pending-invoices?company=...&item_code=...`**
- Drill-down into specific class
- Lists all pending (outstanding > 0) Sales Invoices
- Fields: name, customer, customer_name, item_code, outstanding_amount, grand_total, due_date, status
- Ordered by outstanding_amount DESC

## 6. Payment Workflow (Razorpay)

### End-to-End Flow

```
┌─────────────┐     ┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│  Parent sees │     │  Frontend   │     │   Backend    │     │   Frappe     │
│  outstanding │────▶│  Razorpay   │────▶│   Verify &   │────▶│  Payment    │
│  invoice     │     │  modal      │     │   Record     │     │  Entry       │
└──────┬──────┘     └──────┬──────┘     └──────┬───────┘     └──────┬───────┘
       │                   │                   │                    │
   "Pay Now"          Create Order         Verify HMAC          Reduce
   button              via API             SHA256 sig           outstanding
                                                                on Invoice
```

### Step 1: Create Order (`POST /api/payments/create-order`)

```json
// Request
{ "amount": 13400, "invoice_id": "ACC-SINV-2026-00001", "student_name": "John" }

// Backend logic:
// 1. Validate session cookie (parent must be logged in)
// 2. Convert amount: ₹13,400 → 1,340,000 paise (×100)
// 3. Create Razorpay order via SDK
// 4. Return: { order_id, amount, key_id }
```

### Step 2: Razorpay Modal (`RazorpayPayButton.tsx`)

```
1. Load Razorpay checkout.js script dynamically
2. Open modal with order details (prefilled parent name, email)
3. Parent completes payment (UPI / Card / NetBanking)
4. Razorpay returns: { razorpay_payment_id, razorpay_order_id, razorpay_signature }
```

### Step 3: Verify & Record (`POST /api/payments/verify`)

```json
// Request
{
  "razorpay_payment_id": "pay_xxx",
  "razorpay_order_id": "order_xxx",
  "razorpay_signature": "hmac_xxx",
  "invoice_id": "ACC-SINV-2026-00001",
  "student_name": "John",
  "doctype": "Sales Invoice"
}

// Backend logic:
// 1. HMAC SHA256 verification (key_secret + order_id|payment_id)
// 2. Call Frappe's get_payment_entry whitelisted method
//    → Auto-resolves: debit_to, credit_to, bank account, exchange rate
// 3. Set: reference_no = payment_id, remarks = order + student info
// 4. Submit Payment Entry (docstatus=1)
// 5. Result: Sales Invoice outstanding_amount decreases
//
// FALLBACK: If Payment Entry creation fails:
//   → Post Comment on the Sales Invoice/Fees doctype
//   → Admin can manually reconcile
```

### Step 4: Cache Refresh

```
On success → React Query invalidates "parent-data" cache
→ Parent dashboard refreshes → sees updated outstanding amount
```

### Payment Entry Fields

| Field | Value |
|-------|-------|
| `payment_type` | "Receive" |
| `party_type` | "Customer" (mapped from Student) |
| `party` | Customer name |
| `paid_amount` | Invoice outstanding amount |
| `reference_no` | Razorpay payment ID |
| `references[0].reference_doctype` | "Sales Invoice" |
| `references[0].reference_name` | Invoice name |
| `references[0].allocated_amount` | Paid amount |
| `docstatus` | 1 (auto-submitted) |

## 7. Instalment & Plan System

### Fee Plans

Configured per Fee Structure via custom fields:

| Plan | Description |
|------|-------------|
| **Basic** | Standard fee tier |
| **Intermediate** | Mid-range with additional services |
| **Advanced** | Premium tier (e.g., 1-to-1 tuition) |

### Instalment Options

| Option | Meaning |
|--------|---------|
| `1` | Full payment (single Sales Order) |
| `4` | Quarterly instalments |
| `6` | Bi-monthly instalments |
| `8` | Monthly instalments |

### How Instalments Work

1. **Fee Structure** defines: `custom_plan` + `custom_no_of_instalments` + `total_amount`
2. **During Admission** (`admitStudent`): 
   - Resolves Fee Structure for the student's program + year + branch
   - Creates Sales Order with `custom_plan`, `custom_no_of_instalments` set
3. **Fee Schedule** (batch-level): Can generate multiple Sales Orders/Invoices for a Student Group
4. **Invoicing**: Sales Invoices are created per instalment period from the Sales Order
   - Each invoice = `total / no_of_instalments` (approximate, managed by backend)
5. **Tracking**: `per_billed` on Sales Order tracks how much has been invoiced

### Branch-Wise Fee Lookup

```
Program + Academic Year + Company (Branch) → Fee Structure
  → custom_plan + custom_no_of_instalments → pricing tier
  → Items from Price List (e.g., "10th Tuition Fee" = ₹13,400)
```

## 8. Student Admission & Enrollment

### Full Admission Flow (`admitStudent()`)

```
Step 1: Create Guardian
  └─ { guardian_name, email_address, mobile_number }

Step 2: Create Parent User (Frappe)
  └─ POST /api/auth/create-parent-user
  └─ Creates Frappe User with "Parent" role
  └─ Sends welcome email with login credentials

Step 3: Create Student
  └─ { first_name, last_name, student_email_id }
  └─ custom_branch = company name
  └─ custom_branch_abbr = company abbreviation
  └─ custom_srr_id = sequential ID per branch ("001", "002"…)
  └─ Auto-creates Customer record (for billing)

Step 4: Create & Submit Program Enrollment
  └─ Links: Student ↔ Program ↔ Academic Year
  └─ custom_fee_structure = resolved Fee Structure name
  └─ custom_plan + custom_no_of_instalments from Fee Structure
  └─ Auto-creates Course Enrollments (one per course in program)
  └─ Submits (docstatus=1) — required for fees

Step 5: Add Student to Student Group(s)
  └─ Finds existing batch with capacity
  └─ Or creates new batch: "{BRANCH_ABB}-{PROG_ABB}-{YY}-{SEQ}"

Step 6: Create & Submit Sales Order (if fee_structure provided)
  └─ customer = student's Customer record
  └─ items = tuition fee item from Price List
  └─ custom_plan, custom_no_of_instalments from enrollment
  └─ Auto-submitted
```

### Student Data Model

| Field | Source | Purpose |
|-------|--------|---------|
| `name` | Auto: `EDU-STU-YYYY-NNNNN` | Primary key |
| `student_name` | Computed | Display name |
| `student_email_id` | Input | Required identifier |
| `custom_branch` | Set on creation | Company name (e.g., "Smart Up Chullickal") |
| `custom_branch_abbr` | Set on creation | Short code (e.g., "SU CHL") |
| `custom_srr_id` | Auto-sequential | Branch-scoped serial: "001", "002"… |
| `custom_parent_name` | Set on creation | Fallback parent linkage |
| Customer record | Auto-created | For Sales Order/Invoice tracking |

### SRR ID Generation

Sequential per branch. The system queries existing students at the branch, finds the max `custom_srr_id`, increments by 1, and zero-pads to 3 digits.

## 9. Parent Portal & Dashboard

### Data Architecture

Single admin-backed API endpoint: **`GET /api/parent/data`**

```
Session cookie → parse parent email
  └─ Find Guardian(s) by email
      └─ Find Students via "Student Guardian" child table
          └─ For EACH child (parallel Promise.all):
              ├─ Program Enrollments (draft + submitted)
              ├─ Fee Structure (resolved from enrollment or search)
              ├─ Attendance (current month, submitted only)
              ├─ Fees doctype records
              ├─ Sales Orders (via Customer link)
              └─ Sales Invoices (via Customer link)
```

### Fee Data Priority for Display

| Priority | Source | Use |
|----------|--------|-----|
| 1st | Sales Orders | Full fee commitment (total expected) |
| 2nd | Sales Invoices | What's billed (with outstanding tracking) |
| 3rd | Fees doctype | Legacy/fallback |
| 4th | Fee Structure | Template view if nothing else exists |

### Parent Portal Pages

| Page | Route | Features |
|------|-------|----------|
| **Dashboard** | `/dashboard/parent` | Summary cards: children enrolled, attendance %, total fees, outstanding |
| **Children** | `/dashboard/parent/children` | Per-child: class, branch, batch, fees breakdown, attendance % |
| **Attendance** | `/dashboard/parent/attendance` | Month view: present/absent/late counts, calendar |
| **Fees** | `/dashboard/parent/fees` | Invoice table with **"Pay Now"** (Razorpay) per outstanding row |

## 10. Batches & Student Groups

### Naming Convention

```
{BRANCH_ABBR}-{PROGRAM_ABBR}-{YEAR_2DIGIT}-{SEQUENCE}
Example: CHL-10th-25-1
```

### Structure

| Field | Purpose |
|-------|---------|
| `name` | Auto: naming convention |
| `group_based_on` | "Batch" (always) |
| `program` | Link → Program |
| `batch` | Link → Student Batch Name |
| `academic_year` | Link → Academic Year |
| `academic_term` | Link → Academic Term |
| `custom_branch` | Company/Branch |
| `students[]` | Child table: student links |
| `instructors[]` | Child table: instructor links |
| `max_strength` | Capacity (default: MAX_BATCH_CAPACITY = 60) |

### Auto-Assignment During Admission

1. Search for existing Student Group matching program + branch + year
2. Check if `students.length < max_strength`
3. If capacity available → add student to existing group
4. If full → create new group with incremented sequence number

### Instructor-Batch Relationship

```
Instructor doc → instructor_log[] child table
  Each log: { program, course, custom_branch, academic_year }
  
  → Filter Student Groups where program + branch matches
  → Display as "My Batches" on instructor dashboard
```

## 11. Branch-Wise Scoping

### Company = Branch

Each physical branch is a separate **Company** in Frappe:
- `Smart Up Chullickal` (abbr: SU CHL)
- `Smart Up Ernakulam` (abbr: SU ERV)
- `Smart Up` (parent company)

### API Proxy Auto-Filtering

The proxy layer (`/api/proxy/[...path]`) auto-injects company filters on GET requests:

| DocType | Filter Field | Value |
|---------|-------------|-------|
| Student | `custom_branch` | user's `default_company` |
| Employee | `company` | user's `default_company` |
| Student Group | `custom_branch` | user's `default_company` |
| Course Schedule | `custom_branch` | user's `default_company` |
| Student Attendance | `custom_branch` | user's `default_company` |
| Instructor | `custom_company` | user's `default_company` |

**Exceptions**:
- Admin/Director: No auto-filtering (see all branches)
- Instructors with personal API token: Frappe's User Permissions handle scoping natively

### Security Rules

| Rule | Enforcement |
|------|-------------|
| Non-admin GET requests | Auto-filtered by company |
| Instructor write guard | Can't PUT/POST/DELETE on Student Group |
| Company validation | Session's `allowed_companies` checked on fee API routes |
| Admin-only endpoints | `/api/parent/data`, `/api/auth/create-parent-user`, `/api/admin/*` |

## 12. Email & Notifications

### Email Triggers in Frappe

| Trigger | Source | Mechanism |
|---------|--------|-----------|
| **Parent Welcome Email** | `create-parent-user` route | Frappe's `send_welcome_email=True` on User creation |
| **Fee Payment Request** | Fee Schedule | `send_email` flag → Frappe sends email to `contact_email` on Fees record |
| **Send Payment Request** | Fees doctype | `send_payment_request` field → triggers Frappe's payment request email |
| **Forgot Password** | `/api/auth/forgot-password` | Frontend returns generic message (no actual email — security) |

### Frappe's Built-in Email System

Frappe Cloud handles email delivery natively:
- SMTP configured at site level
- Email templates in Frappe
- Payment Request emails include payment links
- Notifications for document events (submit, cancel, etc.)

### Frontend Notifications

| Feature | Implementation |
|---------|---------------|
| **Class Reminders** | `useClassReminders` hook: polls every 2 min for today's course schedules. Color-coded: urgent (<15min, red), in-progress (green), future (blue) |
| **Notification Bell** | `NotificationDropdown` component with unread count badge |
| **Dismiss Persistence** | Zustand `notificationStore` tracks dismissed notification IDs across refetches |

## 13. Frappe Backend API Reference

### REST API Patterns (via Proxy)

```
GET    /api/resource/{DocType}                    → List records
GET    /api/resource/{DocType}/{name}             → Single record
POST   /api/resource/{DocType}                    → Create record
PUT    /api/resource/{DocType}/{name}             → Update record
DELETE /api/resource/{DocType}/{name}             → Delete record
PUT    /api/resource/{DocType}/{name} {docstatus:1} → Submit
```

## Key Insights

1. **Dual Fee System**: The app uses BOTH Frappe Education's `Fees` doctype AND ERPNext's `Sales Invoice` for billing. Sales Invoice is preferred for aggregated reporting; Fees doctype is the legacy/fallback.

2. **Customer = Student**: Every Student auto-gets a Customer record. All billing (Sales Order, Sales Invoice, Payment Entry) goes through the Customer, not the Student directly.

3. **Branch = Company**: Multi-branch scoping is achieved by mapping each physical branch to a separate Frappe Company. The proxy auto-filters queries by the user's `default_company`.

4. **Payment Entry via Frappe Method**: Razorpay payments use Frappe's `get_payment_entry` whitelisted method, which auto-resolves all accounting accounts. This is more reliable than manual Payment Entry creation.

5. **Fee Schedule → Batch Billing**: Fee Schedule can generate invoices for an entire Student Group at once, handling the batch-level billing workflow.

6. **Instalment Tracking**: `custom_no_of_instalments` on Fee Structure and Sales Order tracks how many payment periods. `per_billed` on Sales Order tracks invoice generation progress.

7. **Admin API for Parent Data**: The parent portal uses a single admin-authenticated endpoint (`/api/parent/data`) that aggregates all child data in one call, avoiding N+1 query issues.

8. **Email via Frappe**: All email (welcome, payment request, forgot password) is handled by Frappe's built-in email system. The frontend only triggers it via API flags.
